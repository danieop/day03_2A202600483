## II. Debugging Case Study (10 Points)

- **Problem Description**: During testing, the agent sometimes failed to follow the required tool-calling format. The local phi3 model was more likely to skip the `Action` block, produce malformed arguments, or jump directly to a final answer even when the question required real-time stock data.

- **Log Source**: The structured JSON logs generated by [src/telemetry/logger.py](../../src/telemetry/logger.py) and written to the daily file under `logs/`.

- **Diagnosis**:
  - The root cause was not the tool code itself, but the model's weaker instruction-following ability.
  - Local phi3 was more sensitive to the strict ReAct formatting requirement, especially when the prompt asked for exact `Thought`, `Action`, and `Final Answer` structure.
  - When the model did not emit a clean `Action: tool_name(args)` line, the parser could not execute the tool and the loop either stopped early or returned an ungrounded response.

- **Solution**:
  - I tightened the system prompt in [src/agent/agent.py](../../src/agent/agent.py) so the model is explicitly forced to call a tool before answering any question that needs live data.
  - I also treated structured logging as the main debugging source, because it clearly shows whether the failure happened in prompt following, tool parsing, or tool execution.
  - After comparing providers, I found that API-based models were much more reliable than local phi3 for this lab because they followed the ReAct format more consistently and produced cleaner tool calls.

## III. Personal Insights: Chatbot vs ReAct (10 Points)

1. **Reasoning**: The `Thought` block helps the agent break a problem into explicit steps. For example, comparing two stocks is no longer one vague answer; the agent can first fetch each stock, then compute the difference, and only then finalize the result. A direct chatbot answer usually compresses all of that into one response, which makes hallucination much more likely.

2. **Reliability**: The agent performed worse than the chatbot when the model did not respect the output format or when the question was simple enough that tool use was unnecessary. In practice, the local phi3 model was the least reliable in this lab because it was more likely to drift from the strict prompt, produce broken `Action` syntax, or ignore the Observation loop entirely. By contrast, API models such as Groq/OpenAI/Gemini were noticeably better at following the exact ReAct structure.

3. **Observation**: The Observation step is what makes the loop grounded. Once the agent sees a tool result, the next decision becomes much more concrete, and the model has less room to invent numbers or guess the answer. This is the main advantage of ReAct over a chatbot: the environment pushes the agent back toward evidence.

## IV. Future Improvements (5 Points)

- **Scalability**: Replace the synchronous tool loop with an asynchronous task queue so multiple tool calls can run in parallel when the question requires it.
- **Safety**: Add a supervisor layer that checks whether the planned action is valid before the tool executes, especially for risky or expensive operations.
- **Performance**: Use schema-based tool calling or function calling to reduce parsing errors, and add caching for repeated stock queries to avoid unnecessary API calls.
- **System Growth**: For a larger agent system, I would add retrieval over tool descriptions and a richer telemetry layer so the agent can choose tools more intelligently and support multi-agent workflows.
